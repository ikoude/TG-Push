## 三、代码评审指导体系

### 3.1 评审指导目标

帮助团队成员提升：
- **评审能力**：从"能找到问题"到"能给出最优解决方案"
- **被评审能力**：从"被动接受意见"到"主动反思改进"
- **沟通能力**：用建设性语言提出和接受技术建议

### 3.2 评审指导矩阵

| 评审者级别 | 能给出的指导 | 期望收到的指导 |
|-----------|------------|--------------|
| 初级工程师 | P2/P3 代码风格问题 | P0/P1 安全性、架构性问题 |
| 中级工程师 | P1 正确性、可维护性问题 | P0 架构决策、疑难问题 |
| 高级工程师 | P0 安全性 + 架构性问题 | 团队流程优化、技术方向 |

### 3.3 评审话术指南

#### ✅ 建设性评审（推荐）

```markdown
"这里的逻辑可以优化，建议使用 early return 模式：

当前：
```javascript
function process(data) {
  if (data !== null) {
    if (data.length > 0) {
      // 处理逻辑...
    }
  }
}
```

建议改为：
```javascript
function process(data) {
  if (!data || data.length === 0) return;
  // 处理逻辑...
}
```

好处：减少嵌套层级，更易读。"
```

#### ❌ 破坏性评审（避免）

```markdown
❌ "这段代码写得太烂了"
❌ "谁写的这个函数？根本不能用"
❌ "你应该重写，而不是修复"
```

### 3.4 评审能力成长路径

```
Level 1: 能发现问题
  └─ 知道什么是"不对的"
  
Level 2: 能描述问题
  └─ 能说清楚"哪里不对"和"为什么不对"
  
Level 3: 能给出方案
  └─ 不仅指出问题，还能提供具体的解决方案
  
Level 4: 能权衡取舍
  └─ 能说明不同方案的trade-off，帮助作者做决策
  
Level 5: 能系统性改进
  └─ 能从单个问题延伸到代码规范、架构优化的层面
```

## 四、复杂问题解决方案研讨

### 4.1 研讨机制

| 活动 | 频率 | 时长 | 参与者 | 产出 |
|------|------|------|--------|------|
| **问题攻关会** | 按需 | 1-2 小时 | 相关开发者 + TL | 解决方案 + Action |
| **架构评审会** | 双周 | 1 小时 | 全体开发者 | 架构优化建议 |
| **案例复盘会** | 每月 | 1 小时 | 全体开发者 | 经验总结入库 |

### 4.2 攻关会流程

```
1. 问题提交（提前 24 小时）
   └─ 在 #tech-discussion 频道提交问题描述
   
2. 资料准备
   └─ 问题复现步骤、相关代码、已尝试的方案
   
3. 集体研讨
   ├─ 10min：问题背景介绍
   ├─ 30min：方案讨论（头脑风暴）
   ├─ 20min：方案评估（优缺点对比）
   └─ 10min：确定 Action 项 + 责任人
   
4. 跟进执行
   └─ 下次攻关会或周会跟进结果
```

### 4.3 典型议题参考

| 类别 | 示例议题 | 解决思路 |
|------|---------|---------|
| 性能 | 消息转发延迟高 | 异步批处理、连接池优化 |
| 安全 | XSS 风险修复方案 | 统一转义函数、Content Security Policy |
| 架构 | 前端状态管理重构 | 引入简单状态机或观察者模式 |
| 运维 | 日志体系优化 | 统一日志格式、分级输出 |

## 六、技术债务清理策略

### 6.1 Tech Debt 分类

| 类型 | 描述 | 典型示例 | 清理优先级 |
|------|------|---------|-----------|
| **设计债务** | 架构设计不合理 | 全局状态滥用 | 高 |
| **代码债务** | 低质量代码 | 重复逻辑、过长函数 | 中 |
| **测试债务** | 缺少测试覆盖 | 核心逻辑无测试 | 中 |
| **文档债务** | 文档缺失/过时 | 缺少使用说明 | 低 |

### 6.2 清理节奏

```
┌─────────────────────────────────────────────────────────────┐
│                    Tech Debt 清理节奏                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  日常：快速修复（< 30min）随手清理，不积累                    │
│                                                             │
│  每周：预留 1 小时处理已识别的 Tech Debt                     │
│        优先级：高价值 + 低 effort                            │
│                                                             │
│  每月：一次 Tech Debt 专项清理日                             │
│        集中处理一个领域的债务                                │
│                                                             │
│  每季度：Tech Debt 评估                                      │
│        重新评估积压债务，决定清理或核销                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 清理流程

```
识别 → 评估 → 排期 → 清理 → 验证 → 沉淀

  1. 识别：在代码审查、问题排查中主动发现债务
     └─ 标记：[Tech Debt] 简短描述
  
  2. 评估：评估影响范围和清理成本
     └─ 记录：影响文件数、预计清理时间、优先级
  
  3. 排期：按优先级进入 Tech Debt Backlog
     └─ 责任人 + 计划日期
  
  4. 清理：按编码规范和审查流程实施
     └─ PR 描述标注 [Tech Debt]
  
  5. 验证：确保功能不受影响，审查通过
     └─ 必要的测试
  
  6. 沉淀：更新最佳实践，避免同类债务
     └─ 完善文档或编码规范
```

## 九、结对编程与导师制度

### 9.1 导师制度

| 角色 | 职责 | 要求 | 收益 |
|------|------|------|------|
| **导师（Mentor）** | 指导新人、传授经验、解答疑惑 | L3+、项目经验 > 1 年 | 教学相长、管理经验 |
| **学员（Mentee）** | 主动学习、按时复盘、反馈进展 | 新入职或转岗人员 | 加速成长 |

#### 导师匹配规则
- 优先选择业务方向匹配的导师
- 每位导师同期指导不超过 2 名学员
- 导师-学员关系每 3 个月重新评估

#### 导师工作清单

| 频率 | 内容 |
|------|------|
| 入职第1周 | 环境搭建指导、项目概览讲解 |
| 前 2 周 | 每日站会对接、问题随时答疑 |
| 第 1 个月 | 每周 1 次 1v1 技术辅导 |
| 第 2-3 个月 | 每 2 周 1 次 progress review |
| 第 3 个月末 | 评估期总结、关系调整或结束 |

### 9.2 结对编程

| 场景 | 适用场景 | 时长 | 形式 |
|------|---------|------|------|
| **常规结对** | 复杂功能开发、技术难题攻关 | 2-4 小时 | driver + observer 轮换 |
| **审查前结对** | 大型 PR 审查前的自查 | 30 分钟 | 作者 + Reviewer |
| **迁移重构结对** | 技术债务清理、架构升级 | 半日或 1 日 | 多人协作 |

#### 结对编程流程

```
1. 准备（10min）
   └─ 明确目标、分解任务

2. 第一个 Sprint（45min）
   ├─ Driver：编写代码
   └─ Observer：实时 review、思考优化

3. 休息（5min）
   └─ 换人、换脑

4. 第二个 Sprint（45min）
   └─ 角色互换

5. 回顾（10min）
   └─ 总结经验、记录 Action
```

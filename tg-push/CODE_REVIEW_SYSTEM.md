# TG·Push 代码审查执行体系 v1.0

> 配合《CODE_REVIEW_STANDARD.md》使用，本文档聚焦于审查的**执行层面**，包括责任分配、频率、工具、反馈与跟进机制。

---

## 一、审查责任分配

### 1.1 角色矩阵

| 角色 | 代码作者 | 同级 Reviewer | 技术负责人 | 质量门禁 |
|------|---------|---------------|-----------|---------|
| **自检** | ✅ 必须 | - | - | - |
| **PR 创建** | ✅ 必须 | - | - | - |
| **代码审查** | - | ✅ 必须 | ✅ 监督 | - |
| **合并批准** | - | 建议 | ✅ 必须 | - |
| **阻塞标记** | - | ✅ 可操作 | ✅ 可操作 | ✅ 自动 |

### 1.2 职责描述

#### 代码作者
- 按照《CODE_REVIEW_STANDARD.md》完成编码自检
- 确保 PR 描述完整、测试步骤可复现
- 及时响应 Review 意见，按优先级修复问题
- 对 P0/P1 问题必须在当次 PR 中修复

#### 同级 Reviewer
- 每次 PR 必须分配至少 1 名同级开发者审查
- 按照审查清单逐项检查，标注问题级别
- 审查意见需具体（指出文件:行号 + 建议方案）
- 复杂问题可发起讨论，但不影响审查进度

#### 技术负责人（TL）
- 疑难问题决策（架构、方向性调整）
- 协调跨模块变更的审查分工
- 监控团队审查质量和效率
- 定期汇总审查数据，优化流程

### 1.3 责任豁免规则

以下情况可跳过强制审查：
- 仅修改 README / 文档（不涉及代码）
- 紧急 hotfix（需 24 小时内补齐审查）
- 自动化工具的配置变更（如 CI/CD pipeline）

## 二、审查频率规范

### 2.1 响应时效要求

| PR 类型 | Reviewer 响应时间 | 作者修复时间 | 最大总周期 |
|---------|------------------|-------------|-----------|
| 普通 PR | **4 小时** | **24 小时** | 3 天 |
| 大型 PR（>300行） | **8 小时** | **48 小时** | 5 天 |
| 紧急 hotfix | **1 小时** | **4 小时** | 24 小时 |

> 超过最大总周期未完成的 PR，自动降级为草稿状态，需重新评估优先级。

### 2.2 定期同步机制

| 活动 | 频率 | 时长 | 参与者 | 产出 |
|------|------|------|--------|------|
| 审查排期会 | 每周一 10:00 | 15 分钟 | 全体开发者 | 本周 PR 排期、分配 Reviewer |
| 审查总结会 | 每周五 16:00 | 30 分钟 | 全体开发者 | 本周审查问题分类、典型案例分享 |
| 质量回顾会 | 每月末 | 60 分钟 | TL + 核心开发 | 质量问题趋势、流程优化提案 |

## 三、审查工具链

### 3.1 工具矩阵

| 阶段 | 工具 | 用途 | 自动化程度 |
|------|------|------|-----------|
| 语法检查 | `node --check` | JS 语法验证 | ✅ 每次提交前必须 |
| 代码风格 | ESLint | 自动检查代码规范 | ✅ CI/CD 集成 |
| 代码格式化 | Prettier | 统一代码格式 | ✅ 自动修复 |
| 复杂度检查 | ESLint (complexity) | 函数复杂度限制 | ✅ CI/CD 集成 |
| 安全扫描 | 手动检查 | XSS、注入等 | ❌ 人工审查 |
| 性能分析 | Chrome DevTools | 前端性能 | ❌ 按需 |
| 单元测试 | - | 业务逻辑验证 | ❌ 暂未引入 |

### 3.2 CI/CD 集成配置

```yaml
# .github/workflows/review.yml 或集成到现有 CI
name: Code Quality Check

on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
          
      - name: Install dependencies
        run: npm install
        
      - name: Run ESLint
        run: npm run lint
        # ESLint 配置见 .eslintrc.json
        
      - name: Check code format
        run: npm run format:check
        # Prettier 配置见 .prettierrc.json
        
      - name: Syntax check
        run: npm run check:all
```

### 3.3 pre-commit 钩子配置

```bash
# .husky/pre-commit 或 .git/hooks/pre-commit
#!/bin/sh

echo "Running pre-commit checks..."

# 1. ESLint 检查
npm run lint
if [ $? -ne 0 ]; then
  echo "ESLint failed. Please fix errors before committing."
  exit 1
fi

# 2. Prettier 检查
npm run format:check
if [ $? -ne 0 ]; then
  echo "Code format issues detected. Run 'npm run format' to fix."
  exit 1
fi

# 3. 语法检查
npm run check
if [ $? -ne 0 ]; then
  echo "Syntax errors detected."
  exit 1
fi

echo "Pre-commit checks passed!"
```

## 四、审查结果反馈机制

### 4.1 反馈渠道

| 渠道 | 适用场景 | 响应优先级 |
|------|---------|-----------|
| PR 评论 | 常规审查意见、代码讨论 | P1：4小时内回复 |
| Slack/飞书 | 紧急阻塞、紧急问题 | P0：1小时内回复 |
| 审查总结会 | 趋势分析、流程问题 | 周会统一讨论 |
| 匿名反馈表单 | 审查质量投诉、流程优化建议 | 月度汇总 |

### 4.2 审查意见模板

#### 问题评论格式
```markdown
**[P1]** `src/server.js:56`

问题：try-catch 中捕获的 error 未记录日志，线上排查困难

建议：
```javascript
catch (error) {
  logger.error('[API] 创建账户失败', { error: error.message, stack: error.stack });
  return res.status(500).json({ success: false, error: '服务器内部错误' });
}
```

可接受替代方案：
- 使用统一错误处理中间件替代每个路由的 try-catch
```

#### 审查通过模板
```markdown
## ✅ 审查通过

| 维度 | 状态 |
|------|------|
| 安全性 | ✅ 无 P0/P1 安全问题 |
| 正确性 | ✅ 逻辑正确、错误处理完善 |
| 可维护性 | ✅ 命名清晰、结构合理 |
| 代码规范 | ✅ ESLint + Prettier 通过 |

**备注：**
- 建议考虑将 `formatLargeNumber()` 抽取为公共函数，后续类似场景可直接复用
```

### 4.3 审查数据统计

每月统计以下指标：

| 指标 | 计算方式 | 目标值 |
|------|---------|-------|
| PR 审查时长 | 从创建到合并的平均时间 | < 2 天 |
| 审查通过率 | 首次提交即通过的比例 | > 60% |
| P0/P1 问题逃逸率 | 上线后才发现的 P0/P1 问题数 | < 5% |
| 评论响应率 | 4小时内回复的审查意见比例 | > 80% |
| ESLint 警告数 | 每月新引入的 ESLint warnings | < 10 |

## 五、问题跟进流程

### 5.1 问题分级与跟踪

| 级别 | 定义 | 跟踪方式 | 关闭条件 |
|------|------|---------|---------|
| **P0** | 安全漏洞、致命bug、阻断问题 | GitHub Issue 立即创建 | 下个版本必须修复 |
| **P1** | 功能不正确、严重性能问题 | GitHub Issue | 本周内修复或确认延期方案 |
| **P2** | 代码可读性、轻微重复 | GitHub Issue | 计划在次月迭代中清理 |
| **P3** | 代码风格、非阻塞优化 | Tech Debt Backlog | 空闲时逐步处理 |

### 5.2 问题生命周期

```
发现问题 → 创建 Issue → 分配责任人 → 修复 → 验证 → 关闭
    ↓           ↓            ↓          ↓      ↓
  Review时    标记级别    指定人+日期   PR审查   Reviewer确认
  即时创建    紧急程度    定期检查      确认修复
```

### 5.3 Tech Debt 管理

#### 积压工作（Tech Debt Backlog）
- 所有 P2/P3 问题统一进入 Tech Debt Backlog
- 每月最后一周预留 20% 开发时间专门处理 Tech Debt
- 优先处理：高价值（影响范围广）+ 低 effort（容易修复）

#### 债务利率制度
```markdown
| 债务类型 | "利率" | 说明 |
|---------|-------|------|
| 快速修复 | 0% | < 30 分钟可解决，立即清理 |
| 计划清理 | 10%/月 | 已列入月度计划，计入迭代 |
| 追踪延期 | 20%/月 | 因业务压力延后，需定期评估 |
| 危险累积 | 50%/月 | 持续影响开发效率，需专项处理 |
```

## 六、大型 PR 处理策略

### 6.1 PR 拆分原则

| 场景 | 拆分策略 |
|------|---------|
| 超过 300 行变更 | 按功能模块拆分为多个 PR |
| 跨多个子系统 | 按子系统边界拆分为独立 PR |
| 重构 + 功能变更混合 | 先拆重构 PR，再做功能 PR |
| 大规模样式调整 | 独立 PR，避免与逻辑变更混合 |

### 6.2 增量审查支持

对于大型重构，提供增量审查材料：
```markdown
## 增量变更说明

本次重构范围：`A → B`（影响 5 个文件）

变更文件清单：
| 文件 | 变更类型 | 变更行数 |
|------|---------|---------|
| app.js | 重构 | +120 / -85 |
| style.css | 调整 | +30 / -20 |

相比上一版本的增量：
[链接到完整 diff]
```

## 七、审查质量保障

### 7.1 Reviewer 能力要求

| 能力维度 | 最低要求 | 达标途径 |
|---------|---------|---------|
| 项目熟悉度 | 能理解任意模块的代码意图 | 参与项目 > 3 个月 |
| 安全意识 | 能识别常见 Web 安全问题 | 通过安全培训 |
| 代码品味 | 能给出具体可操作的改进建议 | 审查经验 > 20 次 |

### 7.2 审查质量互评

每季度进行一次 Reviewer 互评：
- 作者对 Reviewer 的反馈质量打分（1-5 分）
- 匿名汇总，识别审查中的共性问题
- 优秀 Reviewer 经验分享
